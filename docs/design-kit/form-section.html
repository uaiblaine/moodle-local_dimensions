SCSS Personalizado
.local-dimensions-card { … }
O cabeçalho é o nome da categoria de campos personalizados, não uma string do plugin: o instance_form_definition do core faz addElement('header', 'category_<id>', $categoryname) (customfield/classes/handler.php:775) e o handler do plugin repassa o identificador intocado para o nome sair verbatim (competency_handler.php:166-170). Como o plugin não sobrescreve generate_category_name(), a categoria nasce com o padrão do core — Outros campos — e o admin pode renomeá-la na página de configuração dos campos.
A explicação da seção é a proposta. O core não tem descrição de categoria: descrição é por campo (um static depois do campo) e o plugin passa '' em todos eles — "Description removed to avoid fixed-language issue" —, então hoje nenhuma descrição é renderizada nesta seção.
Não há colorpicker. O core só tem seis tipos de campo (checkbox, date, number, select, text, textarea): custombgcolor e customtextcolor são text (displaysize 50, maxlength 255) e o usuário digita o hex. A amostra ao lado do valor é a proposta — o resto da linha é o que ship a hoje.
Nenhum asterisco: todo campo do plugin é provisionado com 'required' => 0 (helper.php:127). Os obrigatórios do modal — shortname e idnumber, com addRule(… 'required' …) — são campos do core, ficam acima desta seção e não pertencem a ela.
Os quatro campos desenhados são os da área competency. SCSS Personalizado só aparece com enablecustomscss ligado, e Formato de Exibição, Subtítulo do acordeão e afins são só da área lp (helper.php:455-487) — não caem no modal de competência.
